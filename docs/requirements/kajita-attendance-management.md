# 勤怠管理Webアプリケーション — 要求仕様書

## 概要

社内全社員（約100名）が利用する勤怠管理Webアプリケーション。
出退勤の打刻、休憩記録、休暇申請、勤怠修正、月次集計を行う。

## ロール定義

| ロール | 説明 |
|--------|------|
| 一般社員 | 自身の打刻・休暇申請・勤怠確認を行う |
| 上長（課長・部長・事業本部長） | 部下の勤怠修正申請・休暇申請を承認する |
| 人事 | 全社員の月次勤怠集計を閲覧・CSVエクスポートする |
| システム管理者 | 社員アカウント管理、部署マスタ管理、勤務形態設定、祝日カレンダー管理を行う |

## ユーザーストーリー

### US-01: 出退勤打刻

> 社員として、Webブラウザ上のボタンで出勤・退勤を打刻したい。勤務時間を正確に記録するため。

- 打刻方法: Webブラウザからのボタン操作のみ
- 出勤打刻と退勤打刻をそれぞれ記録する
- 打刻時刻はサーバー時刻を使用する
- 同日に出勤を2回押した場合はエラーとし、2回目を受け付けない（二重打刻防止）
- 退勤打刻をせずに翌日を迎えた場合、翌日の出勤打刻時に警告を表示し、前日は「退勤忘れ」として打刻修正申請を促す
- 日跨ぎ勤務は発生しない（全員日中勤務を前提とする）

### US-02: 休憩記録

> 社員として、休憩の開始・終了を記録したい。実際の休憩時間を正確に管理するため。

- デフォルト休憩時間: 12:00〜13:00（1時間）
- 手動で休憩を記録しなかった場合、労働基準法に基づき7時間以上の勤務には1時間の休憩を自動で差し引く
- ユーザーが休憩時間を変更可能
- 追加の休憩も記録可能（回数・時間の制限なし、自己申告）
- 休憩開始を記録したまま休憩終了を記録せずに退勤することはできない（エラーとする）
  - 休憩終了ボタンを押さない限り退勤ボタンは無効

### US-03: 打刻修正

> 社員として、打刻ミスや打刻忘れを修正したい。正しい勤怠記録を残すため。

- 当日中（23:59まで）: 本人が自分で修正可能（回数制限なし）
- 翌日以降: 勤怠修正申請を提出 → 上長が承認すると修正が反映される
- 却下された場合、再申請可能（回数制限なし）

### US-04: 勤怠修正の承認

> 上長として、部下の勤怠修正申請を承認・却下したい。正確な勤怠管理のため。

- 承認フロー: 1段階
- 承認先: 自部署の課長・部長・事業本部長の3名から社員が選択可能
- 上長と部下の関係: 部署所属に基づく（部署の長 = 上長）
- 承認期限: 特に制限なし（放置されたまま）
- 代理承認: 不要（3名から選択可能なので別の上長に申請し直す運用）

### US-05: 休暇申請

> 社員として、有給休暇・半休を申請したい。計画的に休暇を取得するため。

- 休暇種別: 有給休暇、半休
- 半休: 午前/午後の区別なし（0.5日消化として扱う）
- 半休時の所定勤務時間: 所定勤務時間の半分を勤務したものとして扱う
- 申請後、上長の承認が必要
- 承認先: 自部署の課長・部長・事業本部長の3名から選択可能
- 当日申請: 可能（上長承認があれば認める）
- 事後申請（過去日）: 不可
- 承認済み休暇の取消: 取消申請 → 上長承認で取消可能

### US-06: 有給残日数管理

> 社員として、有給休暇の残日数を確認したい。取得計画を立てるため。

- 付与タイミング: 入社日基準で労基法に従い付与（入社半年後に10日、以降勤続年数に応じて増加）
- 繰越: 昨年度の未消化分のみ繰越可能（2年で時効）
- 取得（承認済み）で残日数を減算
- 残日数を画面で確認できる

### US-07: 勤怠履歴確認

> 社員として、自分の月別勤怠履歴を確認したい。勤務状況を把握するため。

- 日別の出勤・退勤・休憩・勤務時間を一覧表示
- 月単位で表示

### US-08: 部下の勤怠一覧確認

> 上長として、部下の勤怠一覧を確認したい。チームの勤怠状況を管理するため。

- 自部署の部下全員分を閲覧可能

### US-09: 月次勤怠集計レポート

> 人事として、全社員の月次勤怠集計を確認したい。労務管理のため。

- 集計項目: 総労働時間、残業時間、有給消化数
- 閲覧権限:
  - 本人: 自分のみ
  - 上長: 部下全員分
  - 人事: 全社員分
- 所定労働時間: 月160時間（全社共通・固定値）
- 残業時間: 月の実労働時間 − 所定労働時間（正の場合）
- 残業の区分: 不要（総残業時間のみ集計。深夜・休日の区別なし）
- フレックスタイム制の残業: 月の総労働時間が所定労働時間を超えた分を残業とする（日ごとの残業は計算しない）

### US-10: CSVエクスポート

> 人事として、勤怠データをCSVでエクスポートしたい。外部システム連携や帳票作成のため。

- 外部システム連携: CSVエクスポートのみ（給与計算システム等との直接連携は不要）

### US-11: 社員アカウント管理

> システム管理者として、社員アカウントを作成・削除したい。入退社に対応するため。

- アカウントロック: ログイン5回失敗でロック、管理者が手動解除

### US-12: 部署マスタ管理

> システム管理者として、部署の作成・編集・削除を行いたい。組織変更に対応するため。

- 部署ごとに勤務形態（固定時間制 / フレックスタイム）を設定
- 部署ごとに所定勤務時間を設定

### US-13: 勤務形態設定

> システム管理者として、勤務形態の詳細を設定・変更したい。就業規則の変更に対応するため。

- 固定時間制: 勤務開始・終了時刻を設定
- フレックスタイム: コアタイム（11:00〜15:00）を設定
- 部署単位で適用

### US-14: 祝日・会社カレンダー管理

> システム管理者として、祝日や会社独自の休日を管理したい。正確な営業日管理のため。

- システム管理者が年次で祝日・会社休日を設定する
- カレンダーに反映し、勤怠管理に使用する

### US-15: 通知機能

> 社員として、申請・承認に関する通知をメールで受け取りたい。対応漏れを防ぐため。

- 通知方法: メール通知
- 通知対象:
  - 申請者: 承認・却下時に通知
  - 承認者: 新規申請時に通知

## 認証・認可

- 認証方式: ID / パスワード
- パスワードはハッシュ化して保存（BCrypt）
- パスワードポリシー: 最低12文字、英大小文字+数字+記号
- アカウントロック: ログイン5回失敗でロック、管理者が手動解除
- セッションタイムアウト: 一定時間操作がない場合に自動ログアウト（時間は別途決定）
- ロールに基づくアクセス制御

## 非機能要件

- 利用者数: 約100名
- プラットフォーム: Webブラウザ（PCのみ。モバイル対応不要）
- 技術スタック: Java Spring Boot（バックエンド）、TypeScript Next.js（フロントエンド）
- 稼働時間: 24時間365日（メンテナンス時間除く）
- データ保持期間: 3年
- 監査ログ: 全変更を記録する（打刻修正・承認の変更履歴: 誰がいつ何を変更したか）
- 初期データ: 新規導入（既存システムからのデータ移行不要）

## 用語集

| 用語 | 定義 |
|------|------|
| 打刻 | 出勤・退勤時刻を記録する行為 |
| 所定労働時間 | 月あたりの基準労働時間（全社共通: 160時間/月） |
| 残業時間 | 月の実労働時間 − 所定労働時間（正の場合） |
| コアタイム | フレックスタイム制で必ず勤務する時間帯（11:00〜15:00） |
| 監査ログ | 打刻修正・承認等の変更履歴（誰がいつ何を変更したか） |
