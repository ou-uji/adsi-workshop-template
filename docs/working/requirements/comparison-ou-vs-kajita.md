# 要求仕様の比較レポート — ou 案 vs 梶田案

> 作成: 2026-07-14 / 対象: `docs/requirements/ou-attendance.md` と `docs/requirements/kajita-attendance-management.md`
> 目的: 2 つの勤怠要求案をキャッチアップし、良し悪しを分析。統合方針を提案する。

---

## 0. 結論（先に）

**2 案は競合していない。狙う「ズームレベル」が違う。**

| | ou 案 | 梶田案 |
|---|-------|--------|
| 実体 | **今日のデモを動かす MVP スコープ** | **本格勤怠システムの To-Be 業務要求** |
| ズーム（SDD） | Day 2 / プロダクト内の実装スコープ | プロダクト全体のビジョン |
| 最適な使い道 | **今日そのまま実装に入る土台** | **「本来あるべき姿」の要求カタログ＝将来の道標** |

👉 **推奨: 梶田案を「プロダクト To-Be 要求」として残し、そこから今日のスコープを ou 案として切り出す**（統合案は §5）。どちらかを捨てる必要はない。

---

## 1. 何が違うか（スケール比較）

| 軸 | ou 案 | 梶田案 |
|----|-------|--------|
| 想定規模 | 小規模デモ | 全社 約100名 |
| ユーザーストーリー | 4 ドメイン（D 認証 / A ユーザー / B 打刻 / C 休暇+承認） | **15 US**（打刻・休憩・修正・承認・休暇・有給残・履歴・集計・CSV・アカウント・部署・勤務形態・カレンダー・通知） |
| ロール | ADMIN / MEMBER（2値） | **4 ロール**（一般 / 上長 / 人事 / システム管理者） |
| 業務ルールの深さ | 浅い（打刻・承認の基本のみ） | **深い**（休憩自動控除・フレックス・有給付与/繰越/時効・所定労働時間・残業） |
| 非機能要件 | ほぼ未記載（デモ前提） | **明記**（稼働24/365・保持3年・監査ログ・PCのみ） |
| 実装ターゲット | **本日・初心者チーム・SageMaker** | 実運用システム（期間言及なし） |

---

## 2. 梶田案の評価

### 👍 強み
- **業務ドメインの解像度が高い**。休憩の自動控除（労基法7h→1h）、フレックスのコアタイム、有給の付与/繰越/2年時効、二重打刻防止、退勤忘れ警告など、**現場の実務ルールを捉えている**。これは ou 案が YAGNI で切った領域で、示唆に富む。
- **ロール定義が現実的**。上長＝部署の長、承認先を3名から選択、人事の集計権限など、組織構造に接地。
- **非機能・運用視点がある**。監査ログ（誰がいつ何を変更）、データ保持3年、アカウントロック（5回失敗）、パスワードポリシー（12文字+記号）。
- **用語集がある**。所定労働時間・残業・コアタイム・監査ログを定義 → チームの共通言語になる（IF 合意の素地）。

### 👎 弱み（今日の文脈で）
- **今日作るには壮大すぎる**。15 US・4ロール・フレックス・有給計算・月次集計・CSV・通知・カレンダーは、初心者混在チームが1日で動かす規模を大きく超える。
- **受け入れ条件が TDD 粒度でない**。「〜できる」の記述はあるが、`@DisplayName` に落ちる Given/When/Then や境界値・エラーケースが薄い → そのまま TDD に入れない。
- **チーム開発の「接続部分」が無い**（スライド ⑥ の核）。Unit 分割・依存関係・共通基盤・IF 先行合意・Flyway 連番の衝突回避が未設計 → 並列分担の設計図がない。
- **前提を断定している箇所が多い**（[Q]/[A] の痕跡なし）。「日跨ぎ勤務は発生しない」「所定160h 固定」「半休は午前午後区別なし」等、**仮定が仕様として確定**していて、確認ゲートを経ていない → 後で覆るリスク。
- **技術/コード構造の方針が浅い**。スタックは書いてあるが（SB+Next）、DAO/DTO・レイヤ・DDD 境界・DB(H2/Flyway) 運用など**どう作るか**が無い。

---

## 3. ou 案の評価

### 👍 強み
- **今日のタイムボックスに接地**。3 ドメイン MVP + 認証、デモゴールが「ログイン→打刻→履歴→休暇1件」と明確。**今日動く**を最優先。
- **SDD プロセスに完全準拠**。要求ゲート①、Unit 分割、**共通基盤先行**、**IF 先行合意**、**Flyway 連番予約(V1〜V4)**、`multi-agent-review` まで工程が繋がっている（スライド ⑥ の実践）。
- **受け入れ条件が TDD 粒度**。二重打刻→409、未ログイン→401、不正な状態遷移エラーなど、そのまま Red テストに落ちる。
- **コード構造まで確定**。レイヤード+ライトDDD、**DTO=record 使う / DAO 使わない**、パッケージ縦割り、DB は H2(workshop)+Flyway、認証は BCrypt 最小構成。
- **[Q]/[A] で仮定を潰している**。仕様上の判断をゲートで人間承認 → 仮定リスクが低い。
- **環境制約を反映**（SageMaker absports・Docker回避の H2）。

### 👎 弱み
- **業務ドメインが浅い**。梶田案が持つ休憩・フレックス・有給計算・残業・修正申請などを**YAGNI で切っている**。デモには妥当だが、「勤怠システムの本質」を薄くしか捉えていない。
- **100名規模の非機能を考えていない**（監査ログ・保持期間・同時実行・権限マトリクス）。デモ前提なので当然だが、将来はギャップになる。
- **用語集がない** → 梶田案から借りると良い。
- **ロールが2値**。実組織の上長/人事/管理者の分離は将来必要。

## 4. 直接対決（同じ観点で並べる）

| 観点 | ou 案 | 梶田案 | 今日の勝ち |
|------|:---:|:---:|:---:|
| 今日デモが動くか | ◎ | △（壮大） | **ou** |
| 業務ドメインの深さ | △ | ◎ | 梶田 |
| 受け入れ条件（TDD 化） | ◎ | △ | **ou** |
| チーム分担/IF/共通基盤 | ◎ | ✗ | **ou** |
| 仮定リスク（[Q]/[A]） | ◎ | △（断定多） | **ou** |
| 非機能・運用 | ✗ | ◎ | 梶田 |
| コード構造/DB 方針 | ◎ | △ | **ou** |
| 将来の To-Be 網羅 | △ | ◎ | 梶田 |

> **今日の実装土台としては ou 案が適合**。ただし梶田案は**将来要求のカタログ**として価値が高く、ou 案の視野の狭さを補える。

---

## 5. 統合の提案（両取り）

**梶田案を捨てず、2 層に整理する。**

1. **梶田案 → `docs/requirements/` に「To-Be プロダクト要求」として残す**
   （＝プロダクト全体のズーム。今日は実装しないが、将来の道標・スコープ判断の基準）
2. **ou 案 → 「今日の実装スコープ（MVP）」として維持**。梶田案から**今日でも安く取り込める要素だけ**を選択的に反映:
   - ✅ **用語集を借用**（打刻・所定労働時間・コアタイム等） — コストほぼ0、共通言語になる
   - ✅ **二重打刻防止**は既に ou 案にある（梶田案の「退勤忘れ警告」は今日はスコープ外でOK）
   - ✅ **監査ログの「思想」だけ意識**（今日は実装しないが、状態変更に who/when を残せる設計余地を持つ）
   - 🚫 フレックス・有給計算・月次集計・CSV・通知・カレンダー・修正申請は **今日は入れない**（YAGNI 堅持。膨らませると本日デモが死ぬ）
3. **将来ロードマップ**（任意）: 梶田案の 15 US を Phase 2 以降の Unit 候補として `docs/units/` にバックログ化。

### この統合が良い理由
- スライド ⑥「**固めすぎない**／変更コストが高い骨格だけ決める」に沿う。梶田案の深い業務ルールは"骨格"ではなく"将来の肉付け" → 今日は骨格(ou案)に集中。
- メンバー（梶田さん）のアウトプットを**否定せず活かす**。並行して出したアイディアを To-Be として正式に位置づける＝チームの士気とオーナーシップに効く。

## 6. 一言サマリ

> **ou 案 = 今日の実装図面。梶田案 = 将来の設計思想。** 今日は ou 案で走り、梶田案は To-Be 要求として残して用語集だけ借りる。どちらも捨てない。
