### 技術スタック（テンプレートで確定済み）

- **Backend**: Java / **Spring Boot 4.x**（情報量・Claude との相性・あなたのレビュー力が活きる → 初心者チームに最適）
- **Frontend**: Next.js / TypeScript
- **DB**: 開発は **H2（workshop モード, Docker 不要）** / マイグレーションは **Flyway**
- **アーキ**: レイヤード + ライト DDD（Controller → Service(interface) → Repository(interface) → Entity）
- **起動**: `npm run dev:sagemaker`（ビルド + backend:8080 + frontend + プロキシ一括）
  - プレビュー: PORTS タブの地球儀 → URL の `ports` を **`absports`** に置換（→ `sagemaker-code-editor` スキル）

> 選定理由の詳細（Spring Boot 一択・Quarkus 次点）はあなたの事前検討どおり。ここでは確定事項として扱う。

### コード構造・DAO/DTO 方針（2026-07-14 決定）

repo 規約（`.claude/rules/java-spring-boot.md`）に準拠し、要求レベルで採用パターンを確定：

- レイヤード + ライト DDD：`Controller → Service(interface) → Repository(interface) → Entity`
- **DAO は使わない** — 古典的 DAO でなく **Spring Data JPA Repository(interface)** に統一
- **DTO は使う** — リクエスト/レスポンスは **record**、Entity と分離、手動マッピング（MapStruct 禁止）
- Entity のみ Lombok 可 / DTO は record / DDL は Flyway
- パッケージはドメイン縦割り（common / employee / attendance / leave）＝ 機能で分担
- 詳細な全 API・ER 図は `design` 工程で正式化（要求では方針まで）

→ 確定版の「アーキテクチャ / コード構造方針」節に記載。

## 2. スコープ（MVP）

本日のデモで「動く」ことを目標に、**3 ドメイン**に絞る。スライド DDD 例（勤怠打刻 / 休暇申請 / ユーザー管理）と一致させ、**3 人が並列実装**しやすい形にする。

| # | ドメイン（機能） | 一言 | 重さ | 想定担当 |
|---|-----------------|------|------|---------|
| A | **ユーザー管理** | 社員の登録・一覧（+ role: ADMIN/MEMBER） | 軽 | メンバー1 |
| B | **勤怠打刻** | 出勤・退勤の打刻と当日/一覧表示 | 中 | メンバー2 |
| C | **休暇申請 + 承認** | 申請・一覧 + ADMIN の承認/却下（状態遷移） | **重** | 経験者（あなた）or ペア |

> [Q5] で承認フローをスコープ内にしたため C が最重量。初心者メンバー構成を踏まえ、C は経験者担当 or ペア推奨。

### スコープ外（YAGNI — 今日は作らない）

- 月次集計 / CSV 出力 / PDF 帳票（Day 2 の午後・別 Unit 候補）
- 打刻の丸め・残業・深夜・法定休日などの複雑な勤怠計算
- 通知（メール・Slack）、監査ログ、多言語
- 本格的な権限管理（RBAC の作り込み）

> スコープ外は「捨てる」ではなく「後回し」。IF を切る位置だけ意識しておく（下記 §6）。

## 3. ユーザーストーリー

書式: **「〜（役割）として、〜（目的）のために、〜（機能）したい」**

### ドメイン A: ユーザー管理

- **US-A1**: 管理者として、勤怠を記録する対象を用意するため、**社員を登録**したい（氏名・メール）。
- **US-A2**: 管理者として、対象を確認するため、**社員一覧を表示**したい。
- **US-A3**（任意）: 管理者として、誤登録を直すため、**社員情報を編集**したい。

### ドメイン B: 勤怠打刻（コア）

- **US-B1**: 社員として、勤務開始を記録するため、**出勤打刻**したい（当日・未打刻のときのみ）。
- **US-B2**: 社員として、勤務終了を記録するため、**退勤打刻**したい（当日・出勤済みのときのみ）。
- **US-B3**: 社員として、自分の勤務状況を確認するため、**自分の打刻履歴（一覧）**を見たい。
- **US-B4**（任意）: 社員として、当日の状態を知るため、**今日の打刻状態（未/出勤中/退勤済）**を見たい。

### ドメイン C: 休暇申請

- **US-C1**: 社員として、休暇を取るため、**休暇を申請**したい（日付・種別・理由）。
- **US-C2**: 社員として、申請状況を確認するため、**自分の休暇申請一覧**を見たい。
- **US-C3**: 管理者（ADMIN）として、休暇を認める/断るため、申請を**承認/却下**したい（状態遷移 PENDING → APPROVED/REJECTED）。**[Q5 決定によりスコープ内]**

## 4. 受け入れ条件（Acceptance Criteria）

代表例。TDD の `@DisplayName` にほぼそのまま落ちる粒度で書く。

**US-B1 出勤打刻**
- [ ] 当日未打刻の社員が打刻すると、出勤時刻が記録される
- [ ] 同一社員が当日 2 回目の出勤打刻をするとエラー（例: 409 / バリデーションメッセージ）
- [ ] 存在しない社員 ID で打刻すると 404

**US-B2 退勤打刻**
- [ ] 出勤済みの社員が退勤打刻すると、退勤時刻が記録される
- [ ] 出勤していない社員が退勤打刻するとエラー

**US-A1 社員登録**
- [ ] 氏名・メール必須。未入力はフィールド単位のバリデーションエラー
- [ ] メール形式が不正ならエラー
- [ ] （要決定）メール重複を許すか → §7 [Q3]

**US-C1 休暇申請**
- [ ] 日付・種別必須。未入力はフィールド単位のバリデーションエラー
- [ ] 申請後、一覧に「申請中（PENDING）」で表示される

**US-C3 休暇の承認/却下（ADMIN のみ）**
- [ ] ADMIN が PENDING の申請を承認すると APPROVED になる
- [ ] ADMIN が PENDING の申請を却下すると REJECTED になる
- [ ] すでに APPROVED/REJECTED の申請を再度承認/却下しようとするとエラー（不正な状態遷移）
- [ ] MEMBER（一般）は承認/却下できない（操作不可 or 403 相当）

> エラーはフィールド名 + メッセージで返す。内部エラー詳細はクライアントに返さない（→ `security.md`）。

## 5. ドメインモデル素案（設計への橋渡し）

確定は `design` 工程。ここでは Unit 分割と IF 合意のための**たたき台**。

| Entity | 主な属性 | 所属ドメイン |
|--------|---------|-------------|
| `Employee` | id, name, email, **role（ADMIN/MEMBER）** | A（共有の土台） |
| `AttendanceRecord` | id, employeeId, workDate, clockInAt, clockOutAt | B |
| `LeaveRequest` | id, employeeId, type, startDate, endDate, reason, **status** | C |

Value Object / Enum 候補:
- `Role`（Enum: `ADMIN` / `MEMBER`）— 承認操作の可否に使う
- `LeaveType`（Enum: 有給 / 欠勤 / 特別 など）
- `LeaveStatus`（Enum: `PENDING` / `APPROVED` / `REJECTED`）— 状態遷移を持つ
- 勤務時間 / 時刻レンジ（B の VO 候補）

**共有の核 = `Employee`（＝ employeeId）**。B も C も Employee を参照する（スライドの「共有テーブル」に相当）。

## 6. チーム開発の「つなぎ目」設計（スライド ⑥ の実践）

### 6.1 分割方針

- **機能（ドメイン）で縦に切る**（層で切らない）。A/B/C を 3 人が一気通貫で並列実装。
- 各ドメインは「画面・処理・データ」を内側に完結させ、**作業範囲を重ねない**（DDD 境界）。

### 6.2 共通基盤（**全員・代表 1 台で先に作り切る**）

分担して並列に動く前に、以下を代表 1 台で `main` に用意する（スライド「③ 共通基盤」）:

- [ ] プロジェクト雛形（Spring Boot + Next.js、`npm run setup` が通る）
- [ ] `Employee` の Entity / テーブル（Flyway V1、**role カラム含む**）— **A/B/C 全員が依存する土台**
- [ ] 共有 Enum: `Role`（ADMIN/MEMBER）— A が主管理だが B/C も参照するため土台側に置く
- [ ] 共通の例外ハンドリング（`@RestControllerAdvice`）とエラーレスポンス形式
- [ ] API のベース規約（パス、ページング有無、日付フォーマット、操作者の受け渡し方）
- [ ] アプリ起動・H2・テスト実行（`npm run boot:workshop` / `npm run check:backend`）が回る状態

### 6.3 事前に合意すべきインターフェース（結合でつまずかないために）

スライド「事前に合意すべきインターフェース」に対応。**固めすぎない**（変更コストが高い骨格だけ／変えたら即共有）。

- [ ] **共有データモデル**: `Employee`（id = Long 自動採番, email 一意, **role**）、共有 Enum（`Role` / `LeaveType` / `LeaveStatus`）
- [ ] **API の形状**: パス命名、リクエスト/レスポンス DTO（record）、エラーの返し方、日付/時刻フォーマット（ISO-8601 想定）、**操作者(employeeId/role)の受け渡し方**（認証なしのため）
- [ ] **仕様の範囲**: 例外系・エッジケースをどこまで対応するか（当日重複打刻・不正な状態遷移・権限外の承認操作など）
- [ ] **状態遷移**: 打刻状態（未 → 出勤中 → 退勤済）、休暇ステータス（PENDING → APPROVED/REJECTED、**ADMIN のみ遷移可**）— 誰がいつ変えるか

### 6.4 Unit 依存と Phase（並列化の設計）

```
Phase A（共通基盤・全員1台）  : Employee テーブル + 共通例外 + API規約 + 起動土台
        │
        ├─ Phase B（並列・各自ブランチ）
        │     Unit A: ユーザー管理   （Employee CRUD）   → メンバー1
        │     Unit B: 勤怠打刻       （AttendanceRecord）→ メンバー2
        │     Unit C: 休暇申請       （LeaveRequest）    → メンバー3
        │
        └─ 最後（全員1台）: main に集めて結合 → 通し動作確認 → multi-agent-review
```

- A/B/C は **Employee（Phase A）にのみ依存**し、相互依存はない → **フル並列可能**。
- 結合フェーズで不整合（DB 変更順序の衝突 / DTO 解釈のズレ / コンフリクト）は**必ず起きる**。それは失敗でなく通る道。解消も基本 AI に任せ、採否は人間ゲート。

     推奨: 各ドメイン「一覧 + 入力フォーム」の最小構成。
[Answer]  各ドメイン「一覧 + 入力フォーム」の最小構成。

[Q8] デモの「動いた」の定義（ゴール）
[Answer]  社員を登録 → その社員で出勤/退勤打刻 → 履歴に出る → 休暇を1件申請できる、を

### 決定に伴うスコープ更新（[Q5]/[Q6] の波及）

- **ロール導入**: `Employee.role`（Enum: `ADMIN` / `MEMBER`）を追加。認証はしないため操作者は画面選択、role で操作可否を出し分ける。
- **休暇の状態遷移**: `PENDING → APPROVED / REJECTED`。承認/却下は ADMIN のみ。US-C3 を「任意」から**スコープ内**へ格上げ。
- **担当バランス注意**: ドメイン C（休暇＋承認）が最重量になる → 経験者（あなた）が担当 or 初心者とペア推奨。
